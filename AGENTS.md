# AGENTS.md

This repository is the Apple Text workspace. It ships one Apple text skills collection across multiple surfaces: direct Agent Skills discovery, Claude plugin packaging, a standalone MCP server, and a generated docs site. Keep the package small, predictable, and easy to validate.

## Architecture

Apple Text uses a two-tier delivery model to keep AI context clean:

- **5 registered skills** load inline in Claude Code for routing, decisions, and quick answers
- **4 domain agents** carry a routing table and load the relevant skill file(s) on demand in isolated context — only the focused answer comes back (~3-6k tokens per spawn instead of the full 25k corpus)
- **1 audit agent** (textkit-auditor) scans code for TextKit anti-patterns
- **1 diagnostic agent** (textkit-diagnostics) debugs text symptoms with code inspection
- **MCP server** serves all 39 skills directly for non-Claude clients (VS Code, Cursor, Gemini CLI, etc.)

Domain agents are generated files. Edit the source skill in `skills/*/SKILL.md` and rebuild with `node scripts/build-agents.mjs`. The build script reads `skills/catalog.json` for keyword aliases and detects sidecar files in each skill directory to populate the routing table.

## Structure

- Product surface: `skills/`, `commands/`, `agents/`, `.agents/`, `.claude-plugin/`
- Generated agents: `agents/*-reference.md` (generated from skills by `scripts/build-agents.mjs`)
- MCP surface: `mcp-server/` contains the standalone MCP package, bundle, and server
- Docs surface: `docs/` contains the Astro docs app, generated content, and site config
- Support: `tooling/scripts/`, `tooling/tests/`, `tooling/config/`, `tooling/hooks/`, `tooling/evals/`

## Conventions

- One skill per directory, with the directory name matching the skill `name` in front matter.
- Every skill must have Agent Skills front matter with `name`, `description`, and `license`.
- Skill descriptions should use trigger phrasing such as `Use when...`, not label-style summaries.
- Custom per-skill fields belong under `metadata`, not as extra top-level front matter keys.
- Broad Apple text requests should route through `apple-text` (the router skill).
- Only 5 skills are registered as Skill-tool-invocable: apple-text, apple-text-audit, apple-text-views, apple-text-textkit-diag, apple-text-recipes. All other skills are referenced by domain agents.
- Domain agents (textkit-reference, editor-reference, rich-text-reference, platform-reference) are generated by `node scripts/build-agents.mjs`. Do not edit agent files directly. Each agent carries a routing table and uses Glob/Read to load skill files on demand when spawned.
- When linking from registered skills, use `/skill skill-name` for registered skills and "launch **agent-name** agent" for agent-backed content.
- Prefer focused reference or diagnostic skills over giant catch-all documents.
- Keep examples concrete and Apple-framework specific.

## When Adding a Skill

1. Create `skills/<skill-name>/SKILL.md` with front matter matching the directory name.
2. Add a catalog entry in `skills/catalog.json`.
3. Add it to the appropriate domain agent in `scripts/build-agents.mjs`.
4. Run `node scripts/build-agents.mjs` to regenerate agent files.
5. If the skill should be a registered entry point (rare — only 5 today), add it to `plugin.json` and update the router.
6. Run `npm run check` to validate everything.

## When Changing Packaging

- Keep `.agents/skills` and `.agents/agents` resolving to the source directories so Agent Skills clients can discover the repo directly.
- Use `npm run version:set -- X.Y.Z` or `npm run release -- X.Y.Z` to keep versions aligned. Do not hand-edit version numbers.
- Do not hand-edit generated files (`README.md`, `agents/*-reference.md`, `mcp-server/bundle.json`, `mcp-server/skill-annotations.json`).
- Do not add generated indexes unless they are validated in CI.

## Hooks and Validation

- **pre-commit** (~2s): regenerates agents, docs, MCP artifacts; stages them; runs lint + agents staleness check.
- **pre-push** (~12s): runs full `npm run check` — lint, agents:check, docs:check, plugin validation, MCP staleness, MCP bundle + smoke (12 routing tests + content verification), packaging, description evals, 74 Python tests.
- **CI** (validate.yml): runs `npm run check` on every push and PR.

## Releasing

```bash
npm run release -- X.Y.Z
```

One command: bumps version across all manifests, rebuilds all derived files, runs full validation, commits, tags (`vX.Y.Z`), and pushes. CI then deploys docs and publishes the MCP package.

## Model Selection

All 6 agents use `model: sonnet`. This is a deliberate choice:

- **4 reference agents** discover the plugin's skills directory via Bash, then load 1-2 skill files and extract focused answers — Sonnet handles the Bash→Read→answer loop well.
- **textkit-auditor** scans code for known anti-patterns using Glob/Grep/Read — pattern recognition, not multi-step reasoning.
- **textkit-diagnostics** follows diagnostic decision trees using Glob/Grep/Read — symptom-to-cause mapping, not open-ended reasoning.
- Agent tasks are bounded (max 40 lines output, focused scope) which plays to Sonnet's strengths.
- If complex multi-step debugging questions are routed to agents and quality degrades, consider upgrading the diagnostics agent only.

## Review Standard

Public-facing packaging quality matters here:

- installation should be obvious
- naming should be consistent
- entry points should be clear
- routing should not rely on hidden tribal knowledge
- validation should fail fast when metadata drifts
